Quick Start

CURS3D is a quantum-resistant Layer 1 blockchain written entirely from scratch in Rust (2024 edition). It is not a fork of any existing blockchain. Every component -- consensus, cryptography, networking, storage, smart contracts, fee market, and APIs -- is implemented from zero.

The project has 79 tests passing across 9 modules: api, consensus, core (block, blocktree, chain, transaction, receipt, state_proof), crypto (dilithium, hash), network, storage, vm (gas, state, mod), wallet, and rpc.

Rust edition 2024. Minimum toolchain: stable 1.94+. All on-chain amounts are in microtokens (1 CUR = 1,000,000 microtokens).
terminal
# Clone and build
git clone https://github.com/Pazificateur69/curs3d.git
cd curs3d
cargo build --release

# Create a wallet
./target/release/curs3d wallet --output my_wallet.json

# Start a validator node
./target/release/curs3d node --validator-wallet my_wallet.json

# Run all 79 tests
cargo test

Once the node starts, the following services are available:

  • P2P -- libp2p with Gossipsub + mDNS on port 4337
  • HTTP REST API -- port 8080 (browsers, apps, curl)
  • TCP RPC -- port 9545 (CLI commands like send, stake, status)

State Model

CURS3D uses an account-based state model. Every address has an AccountState:

core/chain.rs -- AccountState
pub struct AccountState {
    pub balance: u64,          // Available balance in microtokens
    pub nonce: u64,            // Transaction counter (replay protection)
    pub staked_balance: u64,   // Locked staking balance in microtokens
    pub public_key: Option<Vec<u8>>,  // Dilithium 5 public key (set on first tx)
}

Addresses

  • Addresses are 20 bytes internally
  • Human-readable format: CUR + 40 hex characters
  • Derived from: SHA-3(public_key)[0..20]

State Root

Every block contains a state root -- the Merkle root computed over all account states and contract storage. Block hashes use double-SHA3: sha3(sha3(bincode(header))).

State Proofs

The state_proof module provides Merkle inclusion proofs for account balances and contract storage values. These proofs allow light clients to verify state without downloading the entire chain.

Smart Contracts: WASM VM

CURS3D runs a deterministic WebAssembly virtual machine powered by Wasmer 5 with the Cranelift compiler backend. Contracts are compiled to WASM and deployed on-chain. The VM uses instruction-level fuel metering to ensure deterministic gas accounting.

Transaction Types

CURS3D supports 6 transaction types:

TypeDescription
TransferSend CUR tokens from one address to another
StakeLock tokens as stake to become a validator (min 1,000 CUR)
UnstakeBegin unstaking tokens (subject to 10-block delay)
CoinbaseBlock reward transaction (50 CUR, created by the block proposer)
DeployContractDeploy WASM bytecode on-chain. The data field contains the bytecode. A new contract address is derived from the deployer address and nonce.
CallContractCall a deployed contract. The to field is the contract address. The data field contains the function selector and arguments.

Contract Storage

Each contract has its own key-value storage namespace managed by the ContractState struct. Keys and values are arbitrary byte arrays. Storage is persisted in the sled database and included in the state root Merkle calculation.

Gas Model

Transaction Receipts

Every transaction that executes produces a Receipt containing execution results:

core/receipt.rs -- Receipt
pub struct Receipt {
    pub status: ReceiptStatus,         // Success or Failure(error_msg)
    pub gas_used: u64,                // Total gas consumed
    pub effective_gas_price: u64,     // Actual price paid per gas unit
    pub priority_fee_paid: u64,       // Priority fee portion
    pub base_fee_burned: u64,         // Base fee burned (EIP-1559)
    pub gas_refunded: u64,            // Unused gas refunded to sender
    pub logs: Vec<LogEntry>,          // Contract log emissions
    pub return_data: Option<Vec<u8>>, // Bytes returned by contract
}

Each LogEntry contains a contract_id (the emitting contract's address) and data (arbitrary bytes). Receipts are stored alongside the block and accessible via the REST API when querying transactions.

Consensus: BFT Proof of Stake

CURS3D uses Byzantine Fault Tolerant Proof of Stake consensus. The protocol guarantees safety as long as fewer than 1/3 of the staked tokens are controlled by malicious actors.

Validator Selection

Validators are selected from the epoch snapshot -- a frozen set of validators captured at epoch boundaries. Selection is stake-weighted and deterministic: given the same block height and epoch snapshot, every node computes the same proposer.

To become a validator, an account must stake at least 1,000 CUR (1,000,000,000 microtokens).

Block Production

  • Blocks are produced every 10 seconds
  • Block reward: 50 CUR (50,000,000 microtokens) per block
  • The proposer creates a Coinbase transaction included in the block
  • Blocks are signed with the proposer's CRYSTALS-Dilithium Level 5 key

Fork Choice

CURS3D uses a BlockTree that tracks all known blocks, including forks. The fork choice rule selects the heaviest chain by cumulative proposer stake.

  • Automatic reorg when a heavier fork is detected
  • Finality boundary prevents reorganizations below finalized height
  • Non-canonical branches are pruned after finalization
  • Common ancestor calculation for efficient chain comparison

Finality

Blocks achieve finality through the FinalityVote and CheckpointVote system managed by the FinalityTracker.

  • Each validator broadcasts a signed FinalityVote for the block they consider canonical
  • When 2/3 of the total stake in the current epoch votes for a block, it becomes finalized
  • Finalized blocks cannot be reverted -- they represent irreversible state
  • Vote deduplication prevents double-counting from the same validator

Provable Slashing

If a validator signs two different blocks at the same height (equivocation), anyone can submit an EquivocationEvidence containing both signed block headers.

  • The evidence is cryptographically verified using the validator's Dilithium public key
  • 33% of the validator's staked balance is slashed (burned)
  • The validator is jailed for 64 blocks (configurable via genesis)
  • Jailed validators cannot propose blocks or cast finality votes during the jail period
  • Slashing evidence is broadcast over the P2P network and persisted in the sled database
Warning: Equivocation is permanent and provable. Once evidence is submitted, the slash cannot be reversed. Validators should never run two instances with the same wallet.

Epochs

The validator set is frozen at epoch boundaries. Each epoch spans 32 blocks (configurable in genesis).

  • At the start of each epoch, an EpochSnapshot captures the active validators and their stakes
  • All block production and finality voting within an epoch uses the snapshot from that epoch's boundary
  • Stake changes (new stakes, unstakes, slashes) take effect at the next epoch boundary
  • This ensures deterministic proposer selection even if stake changes occur mid-epoch

Fee Market: EIP-1559

CURS3D implements an EIP-1559-style dynamic fee market with automatic base fee adjustment, priority fees, and gas refunds.

Fee Fields

Each transaction specifies three fee-related fields:

FieldDescription
max_fee_per_gasMaximum total fee the sender is willing to pay per gas unit
max_priority_fee_per_gasMaximum tip to the block proposer per gas unit
gas_limitMaximum gas the transaction is allowed to consume

Base Fee Adjustment

The base_fee_per_gas is adjusted each block based on how full the previous block was relative to a target (50% of block_gas_limit):

  • If the block used more gas than the target, base fee increases
  • If the block used less gas than the target, base fee decreases
  • The rate of change is controlled by base_fee_change_denominator (default: 8)

Fee Calculation

fee logic
// Effective gas price paid by the sender
effective_gas_price = base_fee_per_gas + priority_fee_per_gas

// Where priority fee is capped:
priority_fee_per_gas = min(max_priority_fee_per_gas, max_fee_per_gas - base_fee_per_gas)

// Base fee portion is burned
base_fee_burned = base_fee_per_gas * gas_used

// Priority fee goes to the block proposer
priority_fee_paid = priority_fee_per_gas * gas_used

// Unused gas is refunded
gas_refunded = (gas_limit - gas_used) * effective_gas_price
The base fee is burned (removed from circulation), creating deflationary pressure. The priority fee incentivizes block proposers to include the transaction. Mempool ordering is by effective priority fee.

Genesis Fee Parameters

ParameterDefaultDescription
block_gas_limit10,000,000Maximum gas per block
initial_base_fee_per_gas1Starting base fee at genesis
base_fee_change_denominator8Controls how quickly base fee adjusts

Network: libp2p

CURS3D uses libp2p 0.54 for peer-to-peer communication with the following components:

  • Gossipsub -- Publish/subscribe protocol for broadcasting blocks, transactions, votes, and slashing evidence
  • mDNS -- Automatic local peer discovery on the same network
  • Bootnodes -- Configurable bootstrap nodes for initial connection to the wider network

The P2P layer listens on port 4337 by default. The network topic is derived from the chain_id and protocol version, ensuring peers on different chains or protocol versions do not interfere.

Network Messages

The following NetworkMessage variants are exchanged over Gossipsub:

MessageDescription
NewBlockBroadcast a newly produced or received block
NewTransactionBroadcast a new transaction to the mempool
RequestBlocksRequest a range of blocks for sync (batches of 50)
BlockResponseResponse containing requested blocks
HeightAnnounceSigned height announcement (every 30s) with protocol version
SlashingEvidenceBroadcast equivocation proof for validator slashing
FinalityVoteValidator finality attestation for a specific block
RequestSnapshotRequest state snapshot manifest for fast sync
SnapshotManifestState snapshot metadata: height, state root, chunk count, chunk hashes
SnapshotChunkIndividual piece of state snapshot with Merkle proof

Sync Protocol

CURS3D supports two sync modes:

Block Sync

  • Blocks are synced in batches of 50
  • Each request has a 15-second timeout with 3 retries
  • Block deduplication prevents processing the same block twice
  • Incompatible peers (different protocol version) are detected via HeightAnnounce and disconnected

State Sync (Fast Sync)

New nodes can fast-sync from a finalized checkpoint instead of replaying the entire chain:

  • The syncing node requests a SnapshotManifest describing the state at a finalized height
  • The manifest contains the height, state root, chunk count, and chunk hashes
  • Each SnapshotChunk is downloaded and verified against its Merkle proof
  • Once all chunks are verified against the state root, the node resumes normal block sync from that height

Storage: sled Database

CURS3D uses sled as its embedded key-value database with schema versioning (currently at v4) and automatic migration support between schema versions.

Persisted Data

  • Blocks -- Indexed by both height and hash for O(1) lookup
  • Account states -- Balance, nonce, staked balance, public key
  • Contract storage -- Per-contract key-value namespaces
  • Pending transactions -- Mempool contents surviving restarts
  • Equivocation evidence -- Persisted slashing proofs
  • Chain metadata -- Height, genesis hash, schema version, genesis config

Schema Versioning

The database schema version is stored in metadata. When a node starts with an older schema version, it automatically migrates to the current version (v4). This ensures forward compatibility across upgrades.

TCP RPC

The TCP RPC server listens on port 9545 (configurable via --rpc-addr) and is used by the CLI commands (send, stake, status). The protocol is a simple newline-delimited JSON exchange over raw TCP.

Protocol Flow

rpc protocol
// Client sends a JSON request (newline-terminated)
{"method": "submit_tx", "params": { ... }}\n

// Server responds with a JSON result (newline-terminated)
{"ok": true, "data": { ... }}\n

The RPC is intended for programmatic use by the CLI and local tooling. For browser/app integration, use the HTTP REST API on port 8080 instead.

Protocol Upgrades

CURS3D supports protocol versioning with upgrade-at-height activation:

  • Each ProtocolUpgrade specifies a height, version, and description
  • Upgrades are defined in the genesis config and activate automatically at the specified block height
  • Nodes running an older protocol version are detected via HeightAnnounce messages and warned
  • The current protocol version is reported in the /api/status response

Docker

CURS3D ships with a multi-stage Dockerfile and a docker-compose configuration for running a 2-node network.

terminal
# Build the Docker image
docker build -t curs3d .

# Run a 2-node network with docker-compose
docker-compose up

# Or run a single node
docker run -p 4337:4337 -p 8080:8080 -p 9545:9545 curs3d \
  curs3d node --validator-wallet /data/wallet.json

The docker-compose setup creates two nodes that automatically discover each other via mDNS and begin syncing. Both nodes expose their HTTP API on port 8080 (mapped to different host ports).
