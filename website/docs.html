Quickstart

CURS3D is a Rust-native Layer 1 with a public node path, REST API, TCP RPC, deployment assets and a repository-level docs stack. The commands below reflect the current binary interface.

terminal 
git clone https://github.com/Pazificateur69/curs3d.git
cd curs3d
cargo build --release

printf '%s\n' 'change-this-validator-password' > validator.password
./target/release/curs3d wallet --output validator.json --password-file validator.password

./target/release/curs3d genesis \
  --output genesis.public-testnet.json \
  --validator-wallet validator.json \
  --validator-password-file validator.password

./target/release/curs3d node \
  --validator-wallet validator.json \
  --validator-password-file validator.password \
  --genesis-config genesis.public-testnet.json

CLI workflow

CommandPurpose
curs3d nodeRun a node with optional validator wallet, public addr, bootnodes and genesis config.
curs3d walletCreate an encrypted wallet file for operational use.
curs3d infoInspect a wallet address and public key.
curs3d sendSubmit a native transfer through the RPC path.
curs3d stakeLock stake for validator participation.
curs3d statusInspect chain status locally or via RPC.
curs3d genesisCreate a public testnet genesis aligned with real wallets.
curs3d bootnode-addressPublish a stable bootnode address for external peers.
The CLI should be understood as an operator-first surface. It is credible when deployment, restart and node identity handling are predictable.

API surface

The HTTP API exposes status, block, account, validator, proof and transaction endpoints. POST routes are used for actions that mutate or submit data.

GET/api/healthzHealth check with chain age and finalized height.

GET/api/block/:heightSingle block view.
GET/api/blocks?from=&limit=Paginated recent blocks.
GET/api/account/:addressBalance, nonce, staked balance.

Token and governance endpoints

The current public surface also includes token registry and governance read endpoints. That matters because these features are now part of the chain state and must remain coherent across restart, replay and reorg.

GET /api/tokens GET /api/token/<address> GET /api/token/<address>/balance/<owner> GET /api/governance/proposals GET /api/governance/proposal/<id>

Operations

CURS3D should be operated as infrastructure, not as a loose experiment. The deployment assets in deploy/ cover public VPS and service management. Wallet passwords, validator material and API tokens must remain outside the repository.

  • P2P defaults to port 4337.
  • HTTP API defaults to 127.0.0.1:8080.
  • TCP RPC defaults to 127.0.0.1:9545.
  • Faucet cooldown persistence is file-backed and should point to a writable operational path.
  • Public APIs should be fronted with normal production hygiene: auth where needed, observability, logs and restart policy.

Current posture

CURS3D has enough surface area to be presented seriously, but it should still speak with precision. The right framing is: substantial protocol and product work, disciplined narrative, and a mainnet-readiness path that still requires hardening.

Credibility comes from alignment: code, explorer, docs, deployment guidance and public claims must describe the same protocol reality.
