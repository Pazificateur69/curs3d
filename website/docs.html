Getting Started

CURS3D is written in Rust and requires the Rust toolchain to build from source. This guide walks you through installing dependencies, compiling the binary, and running your first node.

Prerequisites

  • Rust 1.75+ -- Install via rustup.rs
  • Git -- For cloning the repository
  • A C compiler (gcc or clang) for native dependencies

Install Rust

terminal 
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source $HOME/.cargo/env
rustc --version

Clone the Repository

terminal 
git clone https://github.com/Pazificateur69/curs3d.git
cd curs3d

Building

Build the project in release mode for optimal performance:

terminal 
cargo build --release

The compiled binary will be located at target/release/curs3d. You can optionally install it system-wide:

terminal 
cargo install --path .

Run the test suite (46 tests across 16 modules):

terminal 
cargo test

Running Your First Node

Once built, you can start a node immediately. Without a validator wallet, the node runs in relay mode -- it syncs and relays blocks but does not produce them.

terminal 
# Create a wallet first
curs3d wallet --output my_wallet.json

# Start the node (relay mode)
curs3d node --port 4337 --data-dir curs3d_data

# Start as validator (block production)
curs3d node --port 4337 --validator-wallet my_wallet.json

The node starts both a TCP RPC server on port 9545 and an HTTP REST API on port 8080. Use curs3d status --rpc-addr 127.0.0.1:9545 or curl localhost:8080/api/status to check chain state.

Note: Validator block production requires your wallet to have staked at least 1,000 CUR (the default minimum stake). See the staking command for details.

Docker

CURS3D includes a multi-stage Docker build and a docker-compose configuration for running a 2-node network.

Build the Docker Image

terminal 
docker build -t curs3d .

Run with docker-compose

The included docker-compose.yml launches two connected nodes:

terminal 
docker-compose up -d

# Node 1: port 4337 (libp2p), 9545 (RPC), 8080 (HTTP)
# Node 2: port 4338 (libp2p), 9546 (RPC), 8081 (HTTP)

# Check logs
docker-compose logs -f

# Stop
docker-compose down

Node 2 uses Node 1 as a bootnode and will automatically sync blocks via Gossipsub.

CLI Reference

The curs3d binary provides six subcommands for node operation, wallet management, and chain interaction. Default values are shown below; all flags are optional unless noted.

curs3d node

Starts a CURS3D blockchain node. In relay mode (no validator wallet), the node syncs and gossips blocks. With a validator wallet and sufficient stake, the node produces blocks. The node runs both a TCP RPC server and an HTTP REST API concurrently.

FlagDefaultDescription
--port, -p4337TCP port for libp2p networking
--data-dir, -dcurs3d_dataDirectory for sled database and chain state
--validator-walletNonePath to wallet.json for block production
--bootnodeNoneMultiaddr of peer to connect to (repeatable)
--rpc-addr127.0.0.1:9545Address for the TCP RPC server
--genesis-configNonePath to a custom genesis.json file

Example

terminal 
# Full validator node with bootnode, custom genesis, and both APIs
curs3d node \
  --port 4337 \
  --data-dir ./my_chain \
  --validator-wallet wallet.json \
  --bootnode /ip4/192.168.1.100/tcp/4337 \
  --rpc-addr 0.0.0.0:9545 \
  --genesis-config genesis.json
# HTTP API automatically starts on :8080 (derived from RPC addr)

curs3d wallet

Creates a new CRYSTALS-Dilithium Level 5 keypair and saves it as an AES-256-GCM encrypted wallet file. The wallet password is set interactively and must be at least 8 characters. The key derivation uses Argon2.

FlagDefaultDescription
--output, -owallet.jsonOutput file path for the encrypted wallet
terminal 
curs3d wallet --output validator.json
# Create wallet password: ********
# Confirm password: ********
#
# === CURS3D Wallet Created ===
# Address: CUR7f3a...9c2d
# Saved to: validator.json (AES-256-GCM encrypted)
# Keys: CRYSTALS-Dilithium Level 5

Important: There is no password recovery mechanism. If you lose your password, the wallet and all associated funds are permanently inaccessible.

curs3d info

Displays wallet details including address, public key (truncated), algorithm, and encryption method.

FlagDefaultDescription

terminal
curs3d info --wallet validator.json

# Address:    CUR7f3a...9c2d
# Public Key: a1b2c3d4...
# Algorithm:  CRYSTALS-Dilithium (Level 5)

curs3d send

Sends a transfer transaction. Amount is in whole CUR units (internally multiplied by 1,000,000 for microtokens). Submitted via RPC if a node is running, otherwise falls back to local chain storage.

FlagDefaultDescription
--wallet, -wwallet.jsonSender wallet file
--to, -t(required)Recipient address (hex, with or without CUR prefix)
--amount, -a(required)Amount in CUR (whole units)
--fee, -f1000Transaction fee in microtokens
--data-dircurs3d_dataBlockchain data directory (fallback)
--rpc-addr127.0.0.1:9545RPC server to submit transaction to
terminal
curs3d send \
  --wallet sender.json \
  --to CUR8a4f2b1c3d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9 \
  --amount 100 \
  --fee 1000

curs3d stake

Stakes tokens from your wallet to register as a validator. The minimum stake required is 1,000 CUR (1,000,000,000 microtokens) by default.

FlagDefaultDescription

--amount, -a(required)Amount to stake in CUR (whole units)
--fee, -f1000Transaction fee in microtokens
--data-dircurs3d_dataBlockchain data directory

terminal
curs3d stake --wallet validator.json --amount 1000 --fee 1000

# Stake:     1000 CURS3D
# Fee:       1000 microtokens

# Route:     rpc 127.0.0.1:9545

Note: Unstaking is supported via the Unstake transaction type. The CLI command for unstake will be available in a future release. Currently, unstake transactions can be submitted programmatically via the REST API.

Architecture

CURS3D is organized into eight modules, each responsible for a distinct layer of the protocol stack:

ModuleResponsibility
core/Block, transaction (Transfer, Stake, Unstake, Coinbase), chain state, blocktree (fork choice), Merkle trees, state roots
consensus/Proof of Stake validator selection, finality tracker (2/3 threshold), equivocation evidence, slashing (33% penalty)
crypto/CRYSTALS-Dilithium Level 5 signatures, SHA-3 hashing, double-hash block headers, address derivation
network/libp2p node, Gossipsub messaging, mDNS discovery, bootnode support, signed HeightAnnounce, sync with timeout/retry
storage/sled-backed persistent storage for blocks, accounts, and chain configuration
wallet/Keypair generation, AES-256-GCM encryption with Argon2 key derivation, address formatting
rpc/TCP-based JSON-RPC server for transaction submission, account queries, and status
api/HTTP REST API with 10 endpoints, full CORS support, for browsers and external applications

Consensus Mechanism

CURS3D uses a Proof of Stake consensus with BFT finality. The system is designed around three core properties: liveness, safety, and accountability.

Validator Selection

Validators are accounts with a staked balance meeting or exceeding the minimum stake (default: 1,000 CUR). For each block height, a validator is selected pseudo-randomly weighted by stake:

algorithm
seed = SHA3(block_height || prev_block_hash)
selector = u64_from_le_bytes(seed[0..8]) % total_stake
-- Walk validators sorted by address, subtract stake until selector < stake

BFT Finality

After producing a block, validators broadcast FinalityVote attestations -- Dilithium signatures over block_hash || block_height. A block is finalized when votes from validators representing at least 2/3 of total active stake are collected:

finality check
voted_stake * 3 >= total_active_stake * 2

Once finalized, all non-canonical branches at or below the finalized height are pruned from the blocktree.

Slashing

Equivocation (signing two different blocks at the same height) results in slashing. An EquivocationEvidence contains the block height, validator public key, and two distinct block hashes with valid Dilithium signatures. The penalty is 33% of the validator's staked balance, plus permanent jailing from the validator set.

Fork Choice Rule

CURS3D uses a heaviest chain by cumulative proposer-stake weight fork choice rule. Each block in the BlockTree tracks its cumulative weight. The canonical tip is the block with the highest cumulative weight. Blocks that would fork below the finalized height are rejected.

Transaction Types

CURS3D supports four transaction types:

TypeDescription
TransferMoves tokens from sender to recipient. Debits amount + fee from sender, credits amount to recipient.
StakeMoves tokens from liquid balance to staked balance. Registers the account as a validator candidate.
UnstakeMoves tokens from staked balance back to liquid balance. Reduces validator stake; if below minimum, deactivates validator.
CoinbaseBlock reward transaction. Credits the block reward to the block producer. No sender, no fee, no signature required.

All non-coinbase transactions require a valid Dilithium signature and correct nonce for replay protection.

Cryptography

CRYSTALS-Dilithium Level 5

All signatures use CRYSTALS-Dilithium at NIST security Level 5 (FIPS 204), targeting AES-256 equivalent security. Used for:

  • Transaction signatures
  • Block proposal signatures (double-hash headers)
  • Finality vote attestations
  • Signed height announcements for sync protocol authentication
  • Equivocation evidence verification

SHA-3 (Keccak-256)

Used for block hashing, transaction hashing, Merkle tree construction, state root computation, address derivation, and validator selection seed generation.

Wallet Encryption

Private keys are encrypted with AES-256-GCM. The encryption key is derived from a user password using Argon2 (memory-hard KDF). The wallet file contains the Argon2 salt (16 bytes), AES-GCM nonce (12 bytes), and ciphertext.

Networking

CURS3D uses libp2p with TCP transport, Noise encryption, and Yamux multiplexing.

  • Gossipsub for block and transaction dissemination on a single topic
  • mDNS for local peer discovery
  • Bootnode multiaddresses for remote peer connections
  • Signed HeightAnnounce messages with Dilithium signatures for authenticated sync
  • Sync protocol with 15-second timeout and 3 retries max
  • Block deduplication cache to prevent redundant processing
  • Equivocation detection and automatic evidence broadcast

Storage

CURS3D uses the sled embedded database for persistent storage. Blocks, account states, and chain configuration are stored on disk. Each node can use a separate --data-dir for isolation. The chain survives restarts and can be queried offline via curs3d status --data-dir path.

REST API

The HTTP REST API runs on port 8080 (derived from the RPC address by replacing port 9545 with 8080). All responses use JSON format with CORS headers allowing any origin.

Response format:

JSON response
// Success
{"ok": true, "data": { ... }, "error": null}

// Error
{"ok": false, "data": null, "error": "error message"}

GET /api/status

Returns chain name, height, finalized height, latest hash, genesis hash, pending transactions, and active validator count.

terminal
curl http://localhost:8080/api/status

GET /api/blocks?from=0&limit=20

Returns a list of recent blocks (summaries). Maximum limit is 100. Default returns the latest 20 blocks in reverse order.

terminal
curl "http://localhost:8080/api/blocks?from=0&limit=10"

GET /api/block/:height

Returns full block detail including transactions, state root, merkle root, and validator public key.

terminal
curl http://localhost:8080/api/block/42

GET /api/account/:address

Returns balance, nonce, and staked balance for an address. Accepts hex with or without CUR prefix.

terminal
curl http://localhost:8080/api/account/CUR7f3a8b1c2d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8

GET /api/tx/:hash

Searches all blocks for a transaction matching the given hash. Returns kind, from, to, amount, fee, nonce, and timestamp.

terminal
curl http://localhost:8080/api/tx/a1b2c3d4e5f6...

GET /api/pending

Returns all pending transactions in the mempool.

terminal
curl http://localhost:8080/api/pending

GET /api/validators

Returns the active validator set with addresses, public keys, and stake amounts. Slashed validators are excluded.

terminal
curl http://localhost:8080/api/validators

GET /api/faucet/:address

Testnet faucet. Instantly credits 100 CUR (100,000,000 microtokens) to the given address. No signature required.

terminal
curl http://localhost:8080/api/faucet/CUR7f3a8b1c2d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8

POST /api/tx/submit

Submits a signed transaction as a JSON body. The transaction must include a valid Dilithium signature.

terminal
curl -X POST http://localhost:8080/api/tx/submit \
  -H "Content-Type: application/json" \
  -d '{"kind":"Transfer","from":[...],"sender_public_key":[...],"to":[...],"amount":1000000,"fee":1000,"nonce":0,"timestamp":1234567890,"signature":{"bytes":[...]}}'

TCP RPC

The TCP RPC server runs on port 9545 and uses newline-delimited JSON. It supports three commands:

  • "GetStatus" -- Returns chain status with finalized_height
  • {"GetAccount": {"address": [bytes]}} -- Returns account state
  • {"SubmitTransaction": {"transaction": {...}}} -- Submits a signed transaction
terminal
echo '"GetStatus"' | nc 127.0.0.1 9545

Genesis Config

A genesis configuration file allows you to customize the chain name, block reward, minimum stake, and initial account allocations.

genesis.json
{
  "chain_name": "my-testnet",
  "block_reward": 50000000,
  "minimum_stake": 1000000000,
  "allocations": [
    {
      "public_key": "HEX_ENCODED_DILITHIUM_PUBLIC_KEY",
      "balance": 10000000000,
      "staked_balance": 2000000000
    }
  ]
}

All amounts are in microtokens (1 CUR = 1,000,000 microtokens). The genesis config is validated at node startup. If a chain already exists in the data directory, the genesis config must match or the node will refuse to start.

Block Explorer

CURS3D ships with a built-in Block Explorer web UI. It connects to the REST API and provides:

  • Real-time chain status dashboard (height, finalized height, validators, pending txs)
  • Recent blocks table with expandable transaction details
  • Account lookup (balance, staked balance, nonce)
  • Transaction search by hash
  • Testnet faucet (100 CUR per request)
  • Configurable API endpoint

To use the explorer, start a node and open explorer.html in your browser. Set the API endpoint to http://localhost:8080.
