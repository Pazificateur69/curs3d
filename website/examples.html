Learn by building with CURS3D

Progressive, hands-on guides that take you from running your first node to deploying WASM smart contracts on a quantum-resistant blockchain. Rust 2024 edition, 79 tests, 9 modules.

1 Run Your First Node

Clone the repository, build from source, create a wallet, and start a local devnet node. The entire process takes under two minutes on a modern machine.

terminal
# Clone the repository
git clone https://github.com/Pazificateur69/curs3d.git
cd curs3d

# Build in release mode (Rust 2024 edition)
cargo build --release

# Run the test suite to verify the build (79 tests, 9 modules)
cargo test

Once the build succeeds, create a wallet. Wallets are encrypted with AES-256-GCM using an Argon2-derived key -- you will be prompted for a password.

terminal
# Create an encrypted wallet file
./target/release/curs3d wallet --output my_wallet.json

# View your wallet address and public key
./target/release/curs3d info --wallet my_wallet.json
# Output:
#   Address: CUR7f3a1b2c4d5e6f7890abcdef1234567890abcd
#   Public Key: 0a1b2c3d... (Dilithium 5)

Now start the node. It binds to three ports: P2P on 4337, HTTP API on 8080, and TCP RPC on 9545.

terminal
# Start a validator node
./target/release/curs3d node --validator-wallet my_wallet.json

# Expected output:
# [+] Chain: curs3d-devnet | RPC: 127.0.0.1:9545 | HTTP API: :8080
# [+] P2P listening on /ip4/0.0.0.0/tcp/4337
# [+] Gossipsub active. mDNS discovery enabled.
# [+] Block production every 10 seconds

Your node is now live. The HTTP REST API on port 8080 serves browsers and apps, while the TCP RPC on port 9545 handles CLI commands like curs3d send and curs3d stake. Gossipsub and mDNS peer discovery activate automatically.

2 Send Your First Transaction

Transfer CUR tokens between two wallets using the curs3d send CLI command. This creates a Transfer transaction -- one of the six transaction types supported by the protocol.

terminal
# Create a recipient wallet
./target/release/curs3d wallet --output recipient.json

# Check the recipient's address (CUR + 40 hex chars)
./target/release/curs3d info --wallet recipient.json
# Output: Address: CUR9a8b7c6d5e4f3a2b1c0d9e8f7a6b5c4d3e2f1a

With both wallets ready, send tokens. The fee is specified in microtokens (1 CUR = 1,000,000 microtokens). Make sure your node is running in another terminal.

terminal
# Send 100 CUR to the recipient
./target/release/curs3d send \
  --wallet my_wallet.json \
  --to CUR9a8b7c6d5e4f3a2b1c0d9e8f7a6b5c4d3e2f1a \
  --amount 100 \
  --fee 1000

# The transaction is signed with Dilithium 5, broadcast
# to the network via Gossipsub, and included in the next block.

# Verify the transfer
curl http://localhost:8080/api/account/CUR9a8b7c6d5e4f3a2b1c0d9e8f7a6b5c4d3e2f1a | jq
# {"ok": true, "data": {"balance": 100000000, "nonce": 0, "staked_balance": 0}}

The sender's address is derived from the public key via SHA-3(public_key)[0..20], and displayed in the format CUR + 40 hex characters. The transaction is signed with CRYSTALS-Dilithium Level 5 and verified by every node before inclusion.

3 Become a Validator

Stake CUR tokens to register as a validator and participate in block production. The minimum stake is 1,000 CUR and validators are selected with stake-weighted deterministic selection.

terminal
# Stake the minimum required amount (1000 CUR)
./target/release/curs3d stake \
  --wallet my_wallet.json \
  --amount 1000 \
  --fee 1000

# This creates a Stake transaction that locks your tokens
# and registers you as a validator candidate.

Your validator becomes active at the next epoch boundary. An epoch is 32 blocks, so at the default block time of 10 seconds you will wait at most ~5 minutes. Use the curs3d status command or the REST API to check your progress.

terminal
# Check validator status with the CLI
./target/release/curs3d status

# Or query the REST API for the active validator set
curl http://localhost:8080/api/validators | jq

# Check the current epoch and chain height
curl http://localhost:8080/api/status | jq '.data.epoch'

# Once active, your node proposes blocks and earns
# the block reward of 50 CUR per block (Coinbase tx).

Validators earn the block reward of 50 CUR per proposed block via an automatic Coinbase transaction. Equivocation (signing two different blocks at the same height) results in provable slashing with a 33% stake penalty.

4 Multi-Node Network

Run a local two-node network to test peer discovery, block propagation, and consensus. Each node needs unique ports, a separate data directory, and its own wallet.

terminal -- node 1
# Terminal 1: Start the first node (default ports)
./target/release/curs3d node \
  --port 4337 \
  --data-dir node1_data \
  --validator-wallet wallet1.json \
  --rpc-addr 127.0.0.1:9545

# P2P: 4337 | HTTP API: 8080 | RPC: 9545
terminal -- node 2
# Terminal 2: Start the second node, connecting to the first via bootnode
./target/release/curs3d node \
  --port 4338 \
  --data-dir node2_data \
  --validator-wallet wallet2.json \
  --rpc-addr 127.0.0.1:9546 \
  --bootnode /ip4/127.0.0.1/tcp/4337

# P2P: 4338 | HTTP API: 8081 | RPC: 9546

On the same LAN, mDNS discovery finds peers automatically -- bootnodes are not strictly required but ensure reliable connectivity. Once connected, nodes exchange HeightAnnounce messages every 30 seconds and sync blocks in batches of 50. Blocks propagate via Gossipsub and both nodes participate in consensus.

5 Custom Genesis Configuration

Define your own chain parameters, pre-allocate balances, and schedule protocol upgrades using a genesis JSON file. Every field has a sensible default, so you only need to specify what you want to override.

genesis.json
{
  "chain_id": "my-testnet",
  "chain_name": "My Custom Testnet",
  "block_reward": 100000000,
  "minimum_stake": 500000000,
  "epoch_length": 16,
  "jail_duration_blocks": 32,
  "unstake_delay_blocks": 5,
  "block_gas_limit": 20000000,
  "initial_base_fee_per_gas": 1,
  "base_fee_change_denominator": 8,
  "allocations": [
    {
      "public_key": "<your_hex_pubkey_from_curs3d_info>",
      "balance": 1000000000000,
      "staked_balance": 500000000
    }
  ],
  "upgrades": [
    {
      "height": 500,
      "version": 2,
      "description": "Enable advanced features"
    }
  ]
}

Get your public key hex with curs3d info --wallet my_wallet.json. The allocations array seeds the genesis block with pre-funded accounts. The upgrades array schedules protocol version changes that activate automatically at the specified block heights.

terminal
# Start a node with your custom genesis
./target/release/curs3d node \
  --validator-wallet my_wallet.json \
  --genesis-config genesis.json

# Verify the chain parameters
curl http://localhost:8080/api/status | jq
# "chain_id": "my-testnet"
# "epoch_length": 16
# "block_reward": 100000000

Default values for reference: block reward 50 CUR (50,000,000 microtokens), minimum stake 1,000 CUR (1,000,000,000 microtokens), epoch length 32 blocks, block gas limit 10,000,000.

6 Using the REST API

The HTTP REST API runs on port 8080 with full CORS support. All responses follow the format {"ok": true, "data": {...}} or {"ok": false, "error": "..."}. Use curl, fetch, or any HTTP client.

terminal -- chain status
# Get the full chain status
curl http://localhost:8080/api/status | jq

# Response includes height, finalized height, epoch,
# active validators, protocol version, and chain_id.
# {
#   "ok": true,
#   "data": {
#     "height": 42,
#     "finalized_height": 38,
#     "epoch": 1,
#     "chain_id": "curs3d-devnet",
#     "validator_count": 1
#   }
# }
terminal -- blocks
# Fetch the most recent 20 blocks
curl "http://localhost:8080/api/blocks?limit=20" | jq

# Fetch a specific block by height (genesis = 0)
curl http://localhost:8080/api/block/0 | jq

# The block includes transactions, state_root,
# merkle_root, proposer, and signature.
terminal -- accounts & validators
# Query an account balance (CUR + 40 hex chars)
curl http://localhost:8080/api/account/CUR7f3a1b2c4d5e6f7890abcdef1234567890abcd | jq
# {"ok": true, "data": {"balance": 500000000, "nonce": 3, "staked_balance": 1000000000}}

# List all active validators
curl http://localhost:8080/api/validators | jq

# View pending transactions in the mempool
curl http://localhost:8080/api/pending | jq
terminal -- submit transaction
# Submit a signed transaction
curl -X POST http://localhost:8080/api/tx/submit \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer your_optional_token" \
  -d '{
    "chain_id": "curs3d-devnet",
    "kind": "Transfer",
    "from": [127, 58, 27, ...],
    "sender_public_key": [10, 27, 44, ...],
    "to": [154, 139, 124, ...],
    "amount": 100000000,
    "fee": 1000,
    "nonce": 1,
    "gas_limit": 21000,
    "data": "",
    "timestamp": 1719500000,
    "signature": {"0": [...]},
    "max_fee_per_gas": 100,
    "max_priority_fee_per_gas": 10
  }'

# Response: {"ok": true, "data": {"hash": "a1b2c3..."}}

The optional Authorization: Bearer header enables token-gated access to write endpoints. Max request body size is 1 MB. The six transaction kinds are: Transfer, Stake, Unstake, Coinbase, DeployContract, and CallContract.

7 Unstake Tokens

Withdraw your staked CUR tokens. An unstake delay protects the network from sudden validator exits -- your tokens become available after the delay period elapses.

terminal
# Check your current stake
curl http://localhost:8080/api/account/CUR7f3a1b2c4d5e6f7890abcdef1234567890abcd | jq '.data.staked_balance'
# 1000000000  (1000 CUR in microtokens)

# Submit an Unstake transaction via the CLI
./target/release/curs3d stake \
  --wallet my_wallet.json \
  --amount 1000 \
  --unstake \
  --fee 1000

# This creates an Unstake transaction type.
# The unstake delay is configurable in genesis
# (default: unstake_delay_blocks in genesis.json).
terminal
# Monitor the chain height while waiting for the delay
curl http://localhost:8080/api/status | jq '.data.height'

# After the delay period, check that your tokens are back
curl http://localhost:8080/api/account/CUR7f3a1b2c4d5e6f7890abcdef1234567890abcd | jq
# staked_balance should decrease, balance should increase

# Verify you are no longer in the active validator set
curl http://localhost:8080/api/validators | jq

During the delay, your stake remains locked and you continue to participate in consensus. Once the delay expires, your tokens move from staked_balance back to balance and you are removed from the validator set at the next epoch boundary.

8 Docker Setup

Run CURS3D in Docker containers using the included multi-stage Dockerfile and docker-compose configuration. This is the fastest way to spin up a multi-node network.

terminal
# Build the Docker image (multi-stage: builder + minimal runtime)
docker build -t curs3d .

# Start a pre-configured 2-node network with docker-compose
docker compose up

# Node 1 API: http://localhost:8080
# Node 2 API: http://localhost:8081
# Both nodes discover each other automatically via mDNS.
terminal
# Or run a single node manually with Docker
docker run -p 4337:4337 -p 8080:8080 -p 9545:9545 curs3d \
  curs3d node --validator-wallet /data/wallet.json

# Verify the node is running
curl http://localhost:8080/api/status | jq

# Stop the compose network
docker compose down

The multi-stage Docker build compiles CURS3D in a Rust builder stage and copies only the release binary into a minimal runtime image. The docker-compose file creates wallets, configures bootnodes, and maps ports for both nodes. All three ports are exposed: P2P (4337), HTTP API (8080), and RPC (9545).

9 Block Explorer

Use the built-in web-based block explorer to browse blocks, transactions, account balances, and validator status in real time.

terminal
# Step 1: Make sure a node is running
./target/release/curs3d node --validator-wallet my_wallet.json

# Step 2: Serve the website directory (in another terminal)
cd website
python3 -m http.server 3000

# Step 3: Open the explorer in your browser
# http://localhost:3000/explorer.html

# The explorer connects to http://localhost:8080 by default.
# You can change the API URL in the configuration bar
# at the top of the explorer page.

The block explorer fetches data from the REST API and displays it in a searchable interface. It shows the latest blocks with their transactions, lets you look up any account by address, and displays the active validator set. Data refreshes automatically via polling.

10 Deploy a Smart Contract

Deploy a WebAssembly smart contract to CURS3D. The WASM VM is powered by Wasmer with instruction-level gas metering. Contracts import host functions from the "curs3d" module.

Here is a simple counter contract written in WAT (WebAssembly Text Format) that demonstrates the available host functions:

counter.wat
;; A counter contract for CURS3D
;; Imports host functions from the "curs3d" module

(module
  ;; Host functions provided by the CURS3D WASM VM:
  ;;   storage_get, storage_set     -- key-value storage (i64 keys/values)
  ;;   storage_read, storage_write_bytes -- byte-level storage (ptr+len)
  ;;   emit_log, emit_log_bytes     -- emit contract events
  ;;   consume_gas                  -- manual gas consumption
  ;;   loop_tick                    -- gas metering in loops
  ;;   input, input_len, input_read -- read transaction input data

  (import "curs3d" "storage_get"
    (func $storage_get (param i64) (result i64)))
  (import "curs3d" "storage_set"
    (func $storage_set (param i64 i64)))
  (import "curs3d" "emit_log"
    (func $emit_log (param i64 i64)))
  (import "curs3d" "consume_gas"
    (func $consume_gas (param i64)))

  (memory (export "memory") 1)

  ;; Entry point -- called by the VM
  (func (export "curs3d_call") (result i64)
    (local $count i64)

    ;; Read the current counter from storage slot 0
    (local.set $count
      (call $storage_get (i64.const 0)))

    ;; Increment by 1
    (local.set $count
      (i64.add (local.get $count) (i64.const 1)))

    ;; Write the new value back to storage
    (call $storage_set
      (i64.const 0) (local.get $count))

    ;; Emit a log event (topic=1, value=count)
    (call $emit_log
      (i64.const 1) (local.get $count))

    ;; Return the new counter value
    (local.get $count))
)

Compile the WAT to WASM, hex-encode the bytecode, and deploy via the REST API with a DeployContract transaction.

terminal
# Convert WAT to WASM (requires wabt toolkit)
wat2wasm counter.wat -o counter.wasm

# Hex-encode the WASM bytecode
xxd -p counter.wasm | tr -d '\n' > counter_hex.txt

# Deploy the contract via the REST API
curl -X POST http://localhost:8080/api/tx/submit \
  -H "Content-Type: application/json" \
  -d '{
    "chain_id": "curs3d-devnet",
    "kind": "DeployContract",
    "from": [127, 58, 27, ...],
    "sender_public_key": [10, 27, 44, ...],
    "to": [],
    "amount": 0,
    "fee": 1000,
    "nonce": 1,
    "gas_limit": 500000,
    "data": "'$(cat counter_hex.txt)'",
    "timestamp": 1719500000,
    "signature": {"0": [...]}
  }'

# Gas costs for deployment:
#   Base transaction:     21,000
#   Contract deployment:  32,000
#   Per byte of data:     16
#   Total example:       ~53,200 gas

# The contract address is derived from:
#   SHA-3(deployer_address + nonce)[0..20]

The Wasmer-powered WASM VM validates bytecode structure, injects gas metering at the instruction level, and enforces the block gas limit (10,000,000). Each contract gets isolated key-value storage. The contract address is deterministically computed from the deployer's address and nonce.

11 Call a Smart Contract

Invoke a function on a deployed contract using a CallContract transaction. The VM executes the contract's curs3d_call export and returns a receipt with gas usage, logs, and return data.

terminal
# Call the deployed counter contract
curl -X POST http://localhost:8080/api/tx/submit \
  -H "Content-Type: application/json" \
  -d '{
    "chain_id": "curs3d-devnet",
    "kind": "CallContract",
    "from": [127, 58, 27, ...],
    "sender_public_key": [10, 27, 44, ...],
    "to": [<contract_address_20_bytes>],
    "amount": 0,
    "fee": 1000,
    "nonce": 2,
    "gas_limit": 200000,
    "data": "",
    "timestamp": 1719500000,
    "signature": {"0": [...]}
  }'

# Transaction receipt:
# {
#   "status": "Success",
#   "gas_used": 45832,
#   "logs": [{"topic": 1, "value": 1}],
#   "return_data": "0100000000000000"
# }

Each call to the counter contract increments the stored value and emits a log. The gas costs for contract calls are:

gas costs
# Host function gas costs:
#   storage_get / storage_read     200 gas
#   storage_set / storage_write    5,000 gas
#   emit_log / emit_log_bytes      375 gas
#   consume_gas                    (custom amount)
#   loop_tick                      50 gas
#   host call overhead             40 gas
#
# Transaction-level gas:
#   Base transaction               21,000 gas
#   Contract call overhead         2,600 gas
#   Per byte of input data         16 gas

# Available host functions:
#   storage_get, storage_set
#   storage_read, storage_write_bytes
#   emit_log, emit_log_bytes
#   consume_gas, loop_tick
#   input, input_len, input_read

All storage changes are reflected in the block's state root. Transactions that exceed the remaining block gas budget are deferred to the next block. The EIP-1559 fee market ensures fair pricing with dynamic base fees, priority fees, and automatic gas refunds on unused gas.
