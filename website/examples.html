Tutorials

Code Examples

Hands-on tutorials to get you building with CURS3D. From your first node to multi-validator networks.

Tutorial 1

Run Your First Node

Install CURS3D from source and start a blockchain node on your local machine in under five minutes.

1 Install Rust

CURS3D requires the Rust toolchain. Install it via rustup if you do not have it already.

terminal 
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source $HOME/.cargo/env

2 Clone and Build

Clone the repository and compile in release mode.

terminal 
git clone https://github.com/Pazificateur69/curs3d.git
cd curs3d
cargo build --release

3 Create a Wallet

Generate a Dilithium Level 5 keypair. You will be prompted to set a password.

terminal 
./target/release/curs3d wallet --output my_wallet.json

Expected output:

output 
=== CURS3D Wallet Created ===
Address: CUR7f3a8b1c2d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8
Saved to: my_wallet.json (AES-256-GCM encrypted)
Keys: CRYSTALS-Dilithium Level 5

4 Start the Node

Run a relay node (no block production). The node initializes the chain and listens for peers.

terminal 
./target/release/curs3d node --port 4337 --data-dir node1_data

You will see the CURS3D ASCII banner followed by chain information, PeerId, and RPC address. Press Ctrl+C to stop.

Tutorial 2

Send Your First Transaction

Create two wallets, fund one via genesis allocation, and send tokens between them.

1 Create Two Wallets

terminal 
./target/release/curs3d wallet --output alice.json
./target/release/curs3d wallet --output bob.json

2 Check Alice's Address

terminal 
./target/release/curs3d info --wallet alice.json
# Note the Public Key -- you will need this for genesis allocations

3 Create a Genesis Config

Create a genesis.json that pre-funds Alice's account. Use the hex-encoded public key from the previous step.

genesis.json 
{
  "chain_name": "tutorial-net",
  "block_reward": 50000000,
  "minimum_stake": 1000000000,
  "allocations": [
    {
      "public_key": "PASTE_ALICE_HEX_PUBLIC_KEY_HERE",
      "balance": 10000000000,
      "staked_balance": 2000000000
    }
  ]
}

4 Start a Validator Node

terminal 
./target/release/curs3d node \
  --port 4337 \
  --data-dir tutorial_data \
  --validator-wallet alice.json \
  --genesis-config genesis.json

The node will produce blocks every 10 seconds because Alice's genesis allocation includes staked balance above the minimum.

5 Send Tokens to Bob

Open a second terminal and send 500 CUR from Alice to Bob. Use Bob's address from curs3d info --wallet bob.json.

terminal (new) 
./target/release/curs3d send \
  --wallet alice.json \
  --to BOB_ADDRESS_HERE \
  --amount 500 \
  --fee 1000

The transaction is submitted via RPC to the running node. It will be included in the next block.

6 Verify

terminal 
./target/release/curs3d status --rpc-addr 127.0.0.1:9545

You should see the chain height incrementing and the pending transaction count drop to 0 after inclusion.

Tutorial 3

Become a Validator

Stake tokens to register your node as a block-producing validator on an existing network.

1 Ensure Sufficient Balance

Check your account balance. You need at least 1,000 CUR plus the fee to stake.

terminal 
./target/release/curs3d status --rpc-addr 127.0.0.1:9545

2 Submit Stake Transaction

terminal 
./target/release/curs3d stake \
  --wallet validator.json \
  --amount 1000 \
  --fee 1000

Expected output:

output 
=== CURS3D Stake Submitted ===
Validator: CUR7f3a...9c2d
Stake:     1000 CURS3D
Fee:       1000 microtokens
TxHash:    b5c6d7e8...
Route:     rpc 127.0.0.1:9545

Validator becomes active on-chain after inclusion and once
total stake reaches at least 1000 CURS3D.

3 Run as Validator

Restart your node with the --validator-wallet flag. Once the stake transaction is confirmed, your node will be selected for block production.

terminal 
./target/release/curs3d node \
  --port 4337 \
  --data-dir curs3d_data \
  --validator-wallet validator.json

Look for log messages like Produced block #N to confirm block production.

Tutorial 4

Multi-Node Network

Connect multiple nodes on the same machine and watch them sync blocks via Gossipsub.

1 Start Node A (Validator)

terminal 1 
./target/release/curs3d node \
  --port 4337 \
  --data-dir node_a_data \
  --validator-wallet alice.json \
  --genesis-config genesis.json \
  --rpc-addr 127.0.0.1:9545

Note: On a local network, mDNS will automatically discover peers. For remote nodes, use --bootnode.

2 Start Node B (Relay)

In a second terminal, start another node on a different port with a different data directory. Use the bootnode flag if mDNS does not discover the peer.

terminal 2 
./target/release/curs3d node \
  --port 4338 \
  --data-dir node_b_data \
  --genesis-config genesis.json \
  --rpc-addr 127.0.0.1:9546 \
  --bootnode /ip4/127.0.0.1/tcp/4337

3 Watch Sync

Node B will discover Node A via mDNS or the bootnode address. You will see log messages indicating peer discovery and block synchronization:

node B logs 
Discovered peer: 12D3KooW...
Verified peer 12D3KooW... at height 15 (we: 0). Syncing...
Synced 15 blocks. Height: 15
Accepted block #16 from network

4 Verify Both Nodes

terminal 
# Check Node A
./target/release/curs3d status --rpc-addr 127.0.0.1:9545

# Check Node B
./target/release/curs3d status --rpc-addr 127.0.0.1:9546

Both nodes should report the same chain height and latest hash.

Tutorial 5

Custom Genesis Configuration

Create a genesis file that pre-funds multiple accounts with custom token allocations for a private testnet.

1 Generate Multiple Wallets

terminal 
./target/release/curs3d wallet --output validator1.json
./target/release/curs3d wallet --output validator2.json
./target/release/curs3d wallet --output treasury.json

2 Get Public Keys

Extract the hex-encoded public key from each wallet using curs3d info.

terminal 
./target/release/curs3d info --wallet validator1.json
./target/release/curs3d info --wallet validator2.json
./target/release/curs3d info --wallet treasury.json

3 Write Genesis File

Create a genesis.json with the public keys from step 2. Give each validator enough staked balance to produce blocks and give the treasury a large liquid balance.

genesis.json 
{
  "chain_name": "my-private-testnet",
  "block_reward": 25000000,
  "minimum_stake": 500000000,
  "allocations": [
    {
      "public_key": "VALIDATOR_1_PUBLIC_KEY_HEX",
      "balance": 1000000000,
      "staked_balance": 2000000000
    },
    {
      "public_key": "VALIDATOR_2_PUBLIC_KEY_HEX",
      "balance": 1000000000,
      "staked_balance": 2000000000
    },
    {
      "public_key": "TREASURY_PUBLIC_KEY_HEX",
      "balance": 50000000000,
      "staked_balance": 0
    }
  ]
}

4 Launch the Network

Start both validators with the custom genesis. The block reward is 25 CUR and minimum stake is 500 CUR.

terminal 
# Terminal 1
./target/release/curs3d node --port 4337 --data-dir v1_data \
  --validator-wallet validator1.json --genesis-config genesis.json

# Terminal 2
./target/release/curs3d node --port 4338 --data-dir v2_data \
  --validator-wallet validator2.json --genesis-config genesis.json \
  --bootnode /ip4/127.0.0.1/tcp/4337
Tutorial 6

RPC Integration

Interact with a running CURS3D node programmatically using the TCP-based RPC interface.

1 Ensure a Node is Running

Start a node with the default RPC address 127.0.0.1:9545.

terminal 
./target/release/curs3d node --port 4337 --data-dir rpc_test

2 Query Status via TCP

The RPC protocol is newline-delimited JSON over TCP. You can use netcat or any TCP client.

terminal 
echo '"GetStatus"' | nc 127.0.0.1 9545

Response:

response 
{"Status":{"status":{"chain_name":"curs3d-devnet","height":0,
"finalized_height":0,"latest_hash":"...","genesis_hash":"...",
"pending_transactions":0,"active_validators":0}}}

3 Query Account via Python

Here is a minimal Python script to query an account balance:

query_account.py 
import socket
import json

def rpc_request(addr, port, request):
    with socket.create_connection((addr, port)) as sock:
        payload = json.dumps(request) + "\n"
        sock.sendall(payload.encode())
        response = b""
        while True:
            chunk = sock.recv(4096)
            if not chunk:
                break
            response += chunk
            if b"\n" in response:
                break
    return json.loads(response.decode().strip())

# Query node status
result = rpc_request("127.0.0.1", 9545, "GetStatus")
print(json.dumps(result, indent=2))

# Query a specific account (20 bytes as list of ints)
address_bytes = list(bytes.fromhex("7f3a8b1c2d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a"))
result = rpc_request("127.0.0.1", 9545, {
    "GetAccount": {"address": address_bytes}
})
print(json.dumps(result, indent=2))

4 Submit Transaction via RPC

Transaction submission requires a properly signed Dilithium transaction. The recommended approach is to use the CLI (curs3d send) which handles signing automatically. For programmatic access, you would need to implement Dilithium signing in your client language.

Note: The RPC uses Rust-native serde_json serialization. Transaction fields include byte arrays serialized as JSON arrays of integers. Future versions will include a dedicated SDK for multiple languages.
