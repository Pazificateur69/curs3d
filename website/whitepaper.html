Abstract

We present CURS3D, a Layer 1 blockchain protocol designed to provide long-term security against both classical and quantum adversaries. Written from scratch in Rust 2024, the protocol employs CRYSTALS-Dilithium Level 5 (NIST FIPS 204) for all digital signatures, SHA-3 Keccak-256 for hashing (with double-hash block integrity), and a Byzantine Fault Tolerant Proof of Stake consensus mechanism with deterministic finality at a 2/3 stake-weighted threshold.

Version 3.0 introduces a WASM virtual machine (Wasmer 5 + Cranelift) with instruction-level fuel metering, an EIP-1559 adaptive fee market with base fee burning, epoch-based validator set management (32 blocks per epoch), state sync via Merkle-authenticated snapshot chunks, and a persistent storage layer (sled, schema v4) with 10 purpose-specific trees. The protocol is validated by 79 tests across 9 modules.

This paper specifies the protocol's cryptographic foundations, account model, transaction types, block structure, consensus design with BFT finality and slashing, fork choice rule, fee market, smart contract VM, network protocol, state sync mechanism, API architecture, security guarantees, tokenomics, and development roadmap.

1. Introduction

1.1 The Quantum Threat

The security of existing blockchain protocols rests on the computational hardness of problems such as integer factorization (RSA) and the elliptic curve discrete logarithm problem (ECDLP). These assumptions are vulnerable to quantum algorithms. Shor's algorithm, when executed on a sufficiently powerful quantum computer, can solve both problems in polynomial time, rendering ECDSA, Ed25519, and secp256k1 signatures insecure.

Grover's algorithm provides a quadratic speedup for brute-force search, effectively halving the security level of symmetric ciphers and hash functions. A 256-bit hash provides only 128-bit security against quantum adversaries.

1.2 Design Philosophy

CURS3D addresses this threat by building on NIST-standardized post-quantum algorithms from the outset. Every signature in the protocol uses CRYSTALS-Dilithium, whose security relies on the hardness of the Module Learning With Errors (MLWE) problem over structured lattices, believed to be resistant to both classical and quantum attack. SHA-3 (Keccak-256) provides 128-bit quantum security via Grover's bound.

Rather than retrofitting quantum resistance onto an existing protocol, CURS3D is implemented entirely from scratch -- every component from consensus to networking to storage -- avoiding the technical debt of migration and ensuring a clean, auditable codebase with zero external blockchain dependencies.

2. Cryptographic Primitives

2.1 CRYSTALS-Dilithium Level 5 (NIST FIPS 204)

CURS3D uses CRYSTALS-Dilithium at NIST security Level 5, the highest standardized level. Dilithium provides EUF-CMA (Existential Unforgeability under Chosen Message Attack) security in both the classical and quantum random oracle models. Level 5 provides approximately 256-bit classical security and 128-bit quantum security, with deterministic signatures of approximately 4.6 KB.

Dilithium signatures are used for:

  • Transaction authorization (sender signs serialized transaction data)
  • Block proposal signing (validator signs the double-hashed block header)
  • Finality vote attestation (validator signs block_hash || height || epoch)
  • Height announcement authentication (peers sign network height messages)
  • Equivocation evidence verification (two conflicting signed block headers)

Implementation: pqcrypto-dilithium crate wrapping the reference C implementation.

2.2 SHA-3 (Keccak-256)

SHA-3-256 serves as the universal hash function throughout the protocol:

  • Block hash -- Double-hash construction: sha3(sha3(bincode(header)))
  • Transaction hash -- SHA-3 of bincode-serialized transaction fields
  • Merkle tree -- Binary Merkle tree over transaction hashes
  • State root -- SHA-3 of deterministically sorted and serialized account states
  • Address derivation -- SHA-3(public_key)[0..20]
  • Validator selection seed -- SHA-3(height || prev_hash)
  • Snapshot chunk root -- Merkle root of chunk hashes for state sync verification

The double-hash block construction prevents length-extension attacks and provides an additional security margin against second-preimage attacks.

2.3 Address Format

Addresses are derived deterministically from Dilithium public keys:

address_bytes = SHA-3-256(dilithium_public_key)[0..20]   // 20 bytes
display_form  = "CUR" || hex(address_bytes)               // 43 characters

Example: CUR7f3a8b1c2d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8

Internally, addresses are stored as 20-byte arrays. The CUR prefix is used only in human-readable display contexts (CLI output, API responses, block explorer).

2.4 Wallet Encryption

Private keys are stored encrypted at rest using a layered encryption scheme:

  • KDF -- Argon2 (memory-hard) derives a 256-bit encryption key from a user password
  • Cipher -- AES-256-GCM provides authenticated encryption with associated data
  • Wallet file format -- Argon2 salt (16 bytes) + AES-GCM nonce (12 bytes) + ciphertext

The wallet module supports automatic migration from legacy plaintext wallet files to the encrypted format via load_auto().

3. Account Model

CURS3D uses an account-based state model (as opposed to UTXO). Each account is identified by a 20-byte address and stores the following state:

AccountState {
    balance: u64,                        // Liquid balance in microtokens
    nonce: u64,                          // Monotonically incrementing tx counter
    staked_balance: u64,                 // Tokens locked for validation
    public_key: Option<Vec<u8>>,         // Dilithium public key (set on first tx)
    validator_active_from_height: u64,   // Height at which staking activates
    jailed_until_height: u64,            // Non-zero if validator is jailed
}

All monetary amounts are denominated in microtokens. 1 CUR = 1,000,000 microtokens. The balance type is u64, providing a maximum supply of approximately 18.4 quintillion microtokens (18.4 trillion CUR).

The state root is computed by sorting all accounts deterministically by address, serializing each account state via bincode, and computing a SHA-3-256 hash over the concatenation. An empty state uses the seed b"curs3d-empty-state".

4. Transaction Types

Six transaction types are supported, defined by the TransactionKind enum:

  • Transfer -- Moves tokens from sender to recipient. Requires valid signature, correct nonce, and sufficient balance (amount + fee).
  • Stake -- Moves tokens from liquid balance to staked balance. Activates the sender as a validator if staked balance meets the minimum threshold (default: 1,000 CUR). The sender's public key is registered in the validator set.
  • Unstake -- Moves tokens from staked balance back to liquid balance. Subject to an unstake delay (default: 10 blocks). Deactivates the validator if stake falls below minimum.
  • Coinbase -- Block reward issuance. No sender, no fee, no signature required. Created by the block proposer. Default reward: 50 CUR per block.
  • DeployContract -- Deploys WASM bytecode as a smart contract. The contract address is derived from the deployer address and nonce. Intrinsic gas: GAS_BASE_TX + GAS_DEPLOY = 53,000.
  • CallContract -- Invokes an exported function on a deployed contract. Input data is passed via the transaction data field. Intrinsic gas: GAS_BASE_TX + GAS_CALL = 23,600.

4.1 Transaction Fields

Transaction {
    chain_id: String,                    // Chain identifier (prevents replay across chains)
    kind: TransactionKind,               // One of the six types above
    from: Vec<u8>,                       // 20-byte sender address
    sender_public_key: Vec<u8>,          // Dilithium public key for signature verification
    to: Vec<u8>,                         // Recipient address or contract bytecode
    amount: u64,                         // Value in microtokens
    fee: u64,                            // Legacy fee field
    nonce: u64,                          // Account nonce (must match state)
    timestamp: i64,                      // Unix epoch seconds
    data: Vec<u8>,                       // Arbitrary payload (contract input)
    signature: Option<Signature>,        // Dilithium signature
    gas_limit: u64,                      // Maximum gas units
    max_fee_per_gas: u64,                // EIP-1559: max total fee per gas unit
    max_priority_fee_per_gas: u64,       // EIP-1559: max tip per gas unit
}

4.2 Intrinsic Gas

Every transaction consumes at least its intrinsic gas, computed as:

intrinsic_gas = kind_gas + (payload_bytes * GAS_PER_BYTE)

where kind_gas:
  Transfer / Stake / Unstake / Coinbase = 21,000
  DeployContract                        = 21,000 + 32,000 = 53,000
  CallContract                          = 21,000 +  2,600 = 23,600

5. Block Structure

+──────────────────────────────────────────────────────+
|                     Block Header                      |
|──────────────────────────────────────────────────────|
| version           : u32       Protocol version        |
| height            : u64       Sequential block number  |
| timestamp         : i64       Unix epoch seconds       |
| prev_hash         : [u8; 32]  SHA-3 hash of parent     |
| merkle_root       : [u8; 32]  Merkle root of txs       |
| state_root        : [u8; 32]  SHA-3 of sorted accounts |
| gas_used          : u64       Total gas consumed       |
| base_fee_per_gas  : u64       EIP-1559 base fee        |
| validator_pubkey  : Vec<u8>   Dilithium public key     |
| nonce             : u64       Block nonce              |
+──────────────────────────────────────────────────────+
|  Transactions: Coinbase + Transfer/Stake/Unstake/     |
|                DeployContract/CallContract             |
+──────────────────────────────────────────────────────+
|  block_hash: sha3(sha3(bincode(header)))              |
|  signature:  Dilithium Level 5 over block_hash        |
+──────────────────────────────────────────────────────+

5.1 Block Hash

The block hash is computed via a double-hash construction: sha3(sha3(bincode(header))). The header is first serialized with bincode (deterministic binary encoding), then hashed twice with SHA-3-256. This provides resistance against length-extension attacks and ensures hash commitment integrity.

5.2 Genesis Block

The genesis block has height 0, no signature, a fixed timestamp of 1,700,000,000 (Unix epoch), and a state root derived from the genesis allocations. It is configured via GenesisConfig which specifies chain_name, block_reward, minimum_stake, block_gas_limit, initial_base_fee_per_gas, base_fee_change_denominator, epoch_length, jail_duration_blocks, and initial token allocations.

5.3 State Root

Computed deterministically by sorting all accounts by address (lexicographic byte order), serializing each AccountState via bincode, concatenating, and hashing with SHA-3-256. This ensures all nodes arrive at the same state root for the same account set regardless of processing order.

6. Consensus

6.1 Proof of Stake

Accounts qualify as validators by staking a minimum amount (default: 1,000 CUR = 1,000,000,000 microtokens). The validator set is determined by the current account state -- any account with staked_balance >= minimum_stake and jailed_until_height < current_height is eligible.

6.2 Epoch System

Time is divided into epochs of 32 blocks each. The validator set is frozen at epoch boundaries, meaning stake changes within an epoch do not affect validator selection until the next epoch. Each epoch boundary triggers an EpochSnapshot capturing the validator set, total stake, and chain metadata.

epoch = block_height / epoch_length    // epoch_length = 32

6.3 Validator Selection

Deterministic, stake-weighted, pseudo-random selection for each block height:

seed      = SHA-3(block_height || prev_block_hash)
selector  = u64_from_le_bytes(seed[0..8]) % total_active_stake

// Iterate validators sorted by address (lexicographic)
// First validator where cumulative_stake > selector is selected

This ensures that selection probability is exactly proportional to stake, and any observer can verify the expected proposer for a given height.

6.4 BFT Finality

Validators broadcast FinalityVote attestations -- Dilithium signatures over the concatenation of block_hash || block_height || epoch. The FinalityTracker accumulates votes per block and checks the finality condition:

FinalityVote {
    block_hash:    Vec<u8>,
    block_height:  u64,
    epoch:         u64,
    validator_key: Vec<u8>,
    signature:     Signature,     // Dilithium sign(block_hash || height || epoch)
}

Finality condition:
    voted_stake * 3 >= total_active_stake * 2

Example with 3 validators (3000, 3000, 3000 stake each):
    Total active stake = 9000
    Required: 6000 (2/3 of 9000)
    Vote 1 (3000 stake): 3000 * 3 =  9000 < 18000 -- NOT finalized
    Vote 2 (3000 stake): 6000 * 3 = 18000 >= 18000 -- FINALIZED

Once finalized, a block and all its ancestors become permanent. The BlockTree marks the finalized boundary and prunes all non-canonical branches below that height. Vote deduplication prevents a validator from voting twice for the same block.

6.5 Slashing

Equivocation (signing two different blocks at the same height) is provably detectable. An EquivocationEvidence structure contains:

EquivocationEvidence {
    height:               u64,
    validator_public_key: Vec<u8>,
    block_header_a:       BlockHeader,    // First conflicting header
    block_header_b:       BlockHeader,    // Second conflicting header
    signature_a:          Signature,      // Dilithium signature on header A's hash
    signature_b:          Signature,      // Dilithium signature on header B's hash
}

Verification confirms: (1) the two block hashes differ, (2) both headers commit to the claimed height, (3) both Dilithium signatures verify against the validator's public key. Upon valid evidence:

  • Penalty -- 33% of the validator's staked balance is burned
  • Jailing -- Validator is jailed for 64 blocks (jailed_until_height = current_height + 64)
  • Broadcast -- Evidence is disseminated via Gossipsub to all peers
  • Persistence -- Evidence is stored in the slashing_evidence tree in sled

Automatic equivocation detection occurs when conflicting blocks arrive from the same validator during normal sync. Evidence is constructed and broadcast immediately.

7. Fork Choice

CURS3D resolves competing chain tips using a heaviest chain by cumulative proposer-stake weight rule. The BlockTree maintains a directed acyclic graph (DAG) of all known blocks, where each node stores:

  • Block hash and height
  • Parent hash
  • Proposer's stake at time of proposal
  • Cumulative weight (sum of all proposer stakes from genesis to this block)
Genesis (weight: 0)
  |
  +-- Block A1 (proposer stake: 5000, cumulative: 5000)
  |     |
  |     +-- Block A2 (proposer stake: 3000, cumulative: 8000)  <-- CANONICAL TIP
  |
  +-- Block B1 (proposer stake: 2000, cumulative: 2000)
        |
        +-- Block B2 (proposer stake: 4000, cumulative: 6000)

Chain A wins: 8000 > 6000 cumulative weight

7.1 Finality Boundary

Blocks at or below the finalized height are immutable. The BlockTree rejects insertions below the finality boundary. When a block is finalized via BFT voting, all non-canonical branches at or below that height are pruned, bounding memory overhead.

7.2 Reorganization

When a heavier fork tip is discovered, the chain performs a reorganization: it identifies the common ancestor of the current tip and the new tip, rolls back state to the common ancestor, and reapplies blocks along the heavier fork. This ensures convergence on the globally heaviest chain.

8. Fee Market (EIP-1559)

CURS3D implements an EIP-1559-style adaptive fee market. Each block has a base_fee_per_gas that adjusts automatically based on network utilization.

8.1 Parameters

block_gas_limit             = 10,000,000
target_gas_usage            = block_gas_limit / 2 = 5,000,000
base_fee_change_denominator = 8
initial_base_fee_per_gas    = 0

8.2 Base Fee Adjustment

The base fee for block N+1 is computed from block N:

delta = |parent_gas_used - target_gas|
change = max(parent_base_fee, 1) * delta / (target_gas * denominator)
change = max(change, 1)    // minimum adjustment of 1

if parent_gas_used > target:
    next_base_fee = parent_base_fee + change
else if parent_gas_used < target:
    next_base_fee = parent_base_fee - change
else:
    next_base_fee = parent_base_fee   // no change at target

When parent_base_fee = 0 and gas_used <= target, the base fee remains at 0 (free transactions when the network is underutilized).

8.3 Transaction Fee Fields

max_fee_per_gas          // Maximum total fee per gas unit the sender will pay
max_priority_fee_per_gas // Maximum tip per gas unit (goes to validator)

effective_gas_price = base_fee_per_gas + min(max_priority_fee_per_gas,
                                              max_fee_per_gas - base_fee_per_gas)
priority_fee        = effective_gas_price - base_fee_per_gas

8.4 Fee Distribution

The base fee portion is burned (removed from circulation), creating deflationary pressure proportional to network usage. The priority fee (tip) is paid to the block proposer as an incentive for transaction inclusion.

8.5 Receipts

Every transaction produces a receipt with full fee accounting:

Receipt {
    tx_hash:             Vec<u8>,
    success:             bool,
    gas_used:            u64,
    effective_gas_price: u64,
    priority_fee_paid:   u64,
    base_fee_burned:     u64,
    gas_refunded:        u64,
    logs:                Vec<LogEntry>,
    return_data:         Vec<u8>,
    contract_address:    Option<Vec<u8>>,   // Set on DeployContract
}

9. WASM Virtual Machine

CURS3D executes smart contracts as WebAssembly (WASM) modules using the Wasmer 5 runtime with the Cranelift compiler backend. The VM provides deterministic execution, instruction-level fuel metering, and sandboxed isolation.

9.1 Execution Model

  • Deploy -- WASM bytecode is validated for required loop safety (contracts with loops must import consume_gas or loop_tick). A contract address is derived from the deployer address and nonce. Static gas is estimated by scanning all instructions.
  • Call -- The VM instantiates the contract module, injects host functions, and invokes the exported curs3d_call entry point. Input data is provided via the input() host function.
  • Return -- The WASM return value and any logs emitted via emit_log/emit_log_bytes are captured in the receipt.

9.2 Host Functions

The VM exposes the following host imports under the "curs3d" module namespace:

storage_get(key: i64) -> i64           // Read from contract storage
storage_set(key: i64, value: i64)      // Write to contract storage
storage_read(key_ptr, key_len, dst_ptr, dst_capacity) -> i32  // Byte-level read
storage_write_bytes(key_ptr, key_len, value_ptr, value_len) -> i32
emit_log(topic: i64, data: i64)        // Emit a structured log event
emit_log_bytes(topic_ptr, topic_len, data_ptr, data_len) -> i32
consume_gas(amount: i64)               // Explicit gas consumption
loop_tick(amount: i64)                 // Gas charge for loop iterations
input() -> i64                         // Read transaction input data

9.3 Gas Schedule

Gas costs are defined as constants to ensure deterministic and predictable execution pricing:

GAS_BASE_TX            = 21,000    // Base cost for any transaction
GAS_DEPLOY             = 32,000    // Additional cost for contract deployment
GAS_CALL               =  2,600    // Additional cost for contract call
GAS_STORAGE_READ       =    200    // Read one storage slot
GAS_STORAGE_WRITE      =  5,000    // Write one storage slot
GAS_LOG                =    375    // Emit one log event
GAS_PER_BYTE           =     16    // Per-byte cost for tx payload
GAS_WASM_LOOP_TICK     =     50    // Per-iteration loop gas charge

// Internal WASM instruction costs:
GAS_WASM_DEFAULT_OP    =      2    // Default instruction
GAS_WASM_NUMERIC_OP    =      3    // Arithmetic/comparison
GAS_WASM_CONTROL_OP    =      4    // Control flow (if, br, etc.)
GAS_WASM_MEMORY_OP     =      8    // Memory load/store
GAS_WASM_CALL_OP       =     12    // Function call/return
GAS_HOST_CALL_OVERHEAD =     40    // Overhead per host function call
GAS_MEMORY_READ_BYTE   =      3    // Per-byte host memory read
GAS_MEMORY_WRITE_BYTE  =      6    // Per-byte host memory write

9.4 Loop Safety

To prevent infinite loops, the VM middleware injects consume_gas calls at every loop back-edge and control flow instruction during compilation. Contracts that contain loops but do not import consume_gas or loop_tick are rejected at deployment time. The GAS_WASM_LOOP_TICK charge (50 gas) is automatically added to every loop iteration.

9.5 Contract State

Each contract has isolated key-value storage persisted in the contracts tree in sled. Contract state includes the WASM bytecode, owner address, creation height, and per-contract storage map. Contracts are addressed by their 20-byte derived address.

10. Network Protocol

10.1 Transport Stack

Networking is built on libp2p 0.54 with the following stack:

  • Transport -- TCP
  • Encryption -- Noise protocol (XX handshake pattern)
  • Multiplexing -- Yamux stream multiplexer
  • Identity -- Each node has a unique PeerId derived from its keypair

10.2 Message Dissemination

Gossipsub protocol with strict message validation, 10-second heartbeat interval, and 10 MB maximum message size (to accommodate Dilithium Level 5 signatures at approximately 4.6 KB each).

10.3 Peer Discovery

  • mDNS -- Zero-configuration local network discovery
  • Bootnodes -- Explicit multiaddresses for WAN connectivity

10.4 Network Messages

The protocol defines 10 message types in the NetworkMessage enum:

NewBlock(bincode)              // Broadcast a new block
NewTransaction(bincode)        // Broadcast a pending transaction
RequestBlocks {                // Request a batch of blocks
    from_height, requester_peer_id,
    expected_prev_hash, genesis_hash
}
BlockResponse {                // Response with serialized blocks
    from_height, target_peer_id,
    responder_peer_id, genesis_hash, blocks
}
HeightAnnounce {               // Signed height announcement
    height, latest_hash, genesis_hash, peer_id,
    public_key, signature, protocol_version
}
SlashingEvidence(bincode)      // Equivocation proof
FinalityVote(bincode)          // Validator finality attestation
RequestSnapshot {              // Request state sync snapshot
    requester_peer_id, preferred_height,
    start_chunk, known_finalized_height,
    known_finalized_hash
}
SnapshotManifest {             // Snapshot metadata response
    target_peer_id, data
}
SnapshotChunk {                // Individual snapshot chunk
    target_peer_id, chunk_index, data
}

10.5 Sync Protocol

  • Trigger -- Signed HeightAnnounce messages trigger sync when a peer reports a higher chain height
  • Batch size -- 50 blocks per request
  • Timeout -- 15 seconds per sync request
  • Retries -- 3 maximum retry attempts before marking a peer as unresponsive
  • Validation -- Genesis hash verification prevents cross-chain contamination
  • Deduplication -- Block deduplication cache (1000 entries) prevents redundant processing

10.6 Equivocation Detection

Automatic detection occurs when conflicting blocks arrive from the same validator during normal sync or gossip. Evidence is constructed from the two conflicting signed block headers and broadcast immediately to all peers.

11. State Sync

State sync allows new nodes to bootstrap from a snapshot rather than replaying the entire chain history. The protocol uses Merkle-authenticated chunks for integrity verification.

11.1 Snapshot Manifest

SnapshotManifest {
    height:           u64,
    epoch:            u64,
    chain_id:         String,
    genesis_hash:     Vec<u8>,
    latest_hash:      Vec<u8>,
    tip_height:       u64,
    tip_hash:         Vec<u8>,
    finalized_height: u64,
    finalized_hash:   Vec<u8>,
    state_root:       Vec<u8>,
    chunk_root:       Vec<u8>,       // Merkle root of all chunk hashes
    chunk_count:      usize,
    chunk_hashes:     Vec<Vec<u8>>,  // SHA-3 hash of each chunk
}

11.2 Snapshot State

A full snapshot captures:

  • All blocks (serialized)
  • All account states (address + AccountState pairs)
  • All deployed contracts (address + ContractState pairs)
  • All receipts (hash + Receipt pairs)
  • Pending transactions in the mempool
  • Slashed validator addresses
  • Epoch snapshots (epoch number + EpochSnapshot pairs)
  • Finalized height and hash

11.3 Chunk Verification

Each StateChunk contains its index, data, SHA-3 hash, and a Merkle proof path. The receiving node verifies each chunk's hash against the manifest's chunk_hashes and validates the Merkle proof against the chunk_root. This ensures chunk integrity even when fetched from untrusted peers.

11.4 Sync Flow

1. New node sends RequestSnapshot with preferred_height
2. Peer responds with SnapshotManifest
3. Node validates manifest: genesis_hash match, finalized_height check
4. Node requests chunks starting from start_chunk
5. Each SnapshotChunk is verified against chunk_hashes and proof
6. After all chunks received, state is reconstructed and verified
   against manifest.state_root

13. Security Model

13.1 Quantum Resistance

All digital signatures use CRYSTALS-Dilithium Level 5 (NIST FIPS 204), whose security relies on the MLWE lattice problem. SHA-3 Keccak-256 provides 128-bit quantum security via Grover's bound. No component of the protocol depends on RSA, ECDSA, or any classical-only assumption.

13.2 Byzantine Fault Tolerance

The protocol tolerates f < n/3 Byzantine validators (stake-weighted). The 2/3 supermajority threshold for finality ensures that finalized blocks cannot be reverted as long as at least 2/3 of total stake is controlled by honest validators.

13.3 Equivocation Accountability

Equivocation is cryptographically provable: an EquivocationEvidence containing two valid Dilithium signatures on different block headers at the same height constitutes irrefutable proof of misbehavior. The 33% slash penalty plus 64-block jailing provides a strong economic disincentive.

13.4 Fork Choice Safety

An attacker must control more cumulative proposer stake than the honest chain to produce a heavier fork. Finality prevents both short-range and long-range attacks on finalized blocks. The BlockTree's finality boundary rejects blocks below the finalized height.

13.5 Replay Protection

  • Nonce -- Account nonces must increment monotonically with each transaction. Incorrect nonces are rejected.
  • Chain ID -- Transactions include a chain_id field, preventing replay across different CURS3D networks.
  • Timestamp bounds -- Transactions with timestamps more than 30 seconds in the future or more than 15 minutes old are rejected.

13.6 Block Timestamp Validation

Block timestamps must satisfy: parent.timestamp < block.timestamp <= now + 30 seconds. This prevents timestamp manipulation while allowing reasonable clock skew.

13.7 Wallet Security

Private keys are encrypted at rest with AES-256-GCM using keys derived from user passwords via Argon2 (memory-hard KDF). This protects against offline brute-force attacks on stolen wallet files.

13.8 State Sync Security

Snapshot chunks are verified against Merkle proofs rooted in the chunk_root of the manifest. Genesis hash matching prevents cross-chain state injection. Finalized height validation ensures snapshots are only accepted from sufficiently finalized state.

14. Tokenomics

14.1 Token Unit

Native token: CUR. Smallest unit: microtoken. 1 CUR = 1,000,000 microtokens. All on-chain amounts are stored as u64 microtokens.

14.2 Block Rewards

Default block reward: 50 CUR (50,000,000 microtokens) per block. Issued via a Coinbase transaction to the block proposer. Configurable via genesis configuration.

14.3 Fee Economics

  • Base fee -- Determined algorithmically by EIP-1559 mechanism. Burned on use, creating deflationary pressure.
  • Priority fee -- Optional tip to the block proposer. Paid from the spread between max_fee_per_gas and base_fee_per_gas.
  • Gas refund -- Unused gas (within the gas limit) is refunded to the sender.

14.4 Staking Economics

  • Minimum stake -- 1,000 CUR (1,000,000,000 microtokens), configurable via genesis
  • Selection probability -- Exactly proportional to staked amount
  • Block rewards -- Provide return on staked capital (50 CUR per block to the proposer)
  • Unstake delay -- 10 blocks cooling period before staked tokens return to liquid balance
  • Epoch freezing -- Stake changes take effect at the next epoch boundary (32 blocks)

14.5 Slashing Economics

Equivocation penalty: 33% of staked balance burned + 64-block jailing period. The expected value of equivocation is strongly negative: a validator with 10,000 CUR staked would lose 3,300 CUR and be unable to propose blocks for 64 blocks.

16. Conclusion

CURS3D demonstrates that a blockchain protocol can be built entirely on post-quantum cryptographic primitives without sacrificing modern Layer 1 capabilities: deterministic BFT finality, stake-weighted consensus with provable slashing, adaptive fee markets, smart contract execution, efficient state sync, and production-grade infrastructure.

By adopting CRYSTALS-Dilithium Level 5 and SHA-3 from inception, CURS3D avoids the technical debt of retrofitting quantum resistance onto a classical foundation. The EIP-1559 fee market provides predictable transaction costs with deflationary base fee burning. The WASM virtual machine with instruction-level gas metering enables expressive smart contracts within deterministic resource bounds. The state sync protocol with Merkle-authenticated chunks enables new nodes to join the network efficiently without replaying the full chain history.

Appendix A: Storage Schema

Persistent storage uses sled (embedded key-value database) with schema version 4. Ten purpose-specific trees:

Tree Name               Contents
────────────────────    ────────────────────────────────────────
blocks                  Serialized blocks keyed by height
accounts                AccountState keyed by 20-byte address
meta                    Chain metadata (height, schema version, genesis config)
pending                 Pending mempool transactions
slashing_evidence       EquivocationEvidence records
epochs                  EpochSnapshot keyed by epoch number
contracts               ContractState keyed by contract address
receipts                Transaction receipts keyed by tx hash
snapshot_manifests      SnapshotManifest keyed by height
snapshot_chunks         StateChunk keyed by (height, chunk_index)

The schema includes automatic migration from older versions. The meta tree stores the schema_version key for version detection on startup.
