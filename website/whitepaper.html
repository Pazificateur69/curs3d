Abstract

We present CURS3D, a Layer 1 blockchain protocol designed to provide long-term security against both classical and quantum adversaries. The protocol employs CRYSTALS-Dilithium Level 5 for all digital signatures, SHA-3 for hashing, and a Byzantine Fault Tolerant Proof of Stake consensus mechanism with deterministic finality. The system is implemented in Rust for memory safety and performance, uses libp2p for decentralized peer-to-peer networking, and persists state to a sled embedded database. This paper describes the protocol's cryptographic foundations, consensus design, fork choice rule, network architecture, and security guarantees.

1. Introduction

The security of existing blockchain protocols rests on the computational hardness of problems such as integer factorization (RSA) and the elliptic curve discrete logarithm problem (ECDLP). These assumptions are vulnerable to quantum algorithms. Shor's algorithm, when executed on a sufficiently powerful quantum computer, can solve both problems in polynomial time, rendering ECDSA and EdDSA signatures -- used by virtually all production blockchains -- insecure.

The timeline for cryptographically relevant quantum computers remains uncertain, but the risk is material. Assets secured by classical cryptography today could be retroactively compromised once quantum capability exists. This "harvest now, decrypt later" threat model demands that new blockchain protocols adopt post-quantum cryptographic primitives from the outset rather than retrofitting them later.

CURS3D addresses this threat by building on NIST-standardized post-quantum algorithms. Every signature in the protocol -- from wallet transactions to block proposals to consensus votes -- uses CRYSTALS-Dilithium, a lattice-based scheme whose security relies on the hardness of the Module Learning With Errors (MLWE) problem, which is believed to be resistant to quantum attack.

2. Cryptographic Primitives

2.1 CRYSTALS-Dilithium

CURS3D uses CRYSTALS-Dilithium at NIST security Level 5, the highest security tier specified in FIPS 204. Level 5 targets security equivalent to AES-256, meaning an adversary with access to a quantum computer would need computational resources comparable to breaking AES-256 to forge a signature.

Dilithium is a lattice-based digital signature scheme based on the Fiat-Shamir with Aborts framework. It provides:

  • EUF-CMA security (existential unforgeability under chosen message attack) in both classical and quantum random oracle models
  • Deterministic signatures, eliminating nonce-reuse vulnerabilities
  • Relatively compact signatures compared to other post-quantum schemes (approximately 4.6 KB at Level 5)

Dilithium is used for all signature operations in CURS3D: transaction authorization, block proposal signing, finality vote attestation, and authenticated height announcements in the sync protocol.

2.2 SHA-3 (Keccak)

SHA3-256 serves as the universal hash function across the protocol. It is used for:

  • Block hash computation
  • Transaction hash computation
  • Merkle tree construction for transaction inclusion proofs
  • State root computation over the account set
  • Address derivation from Dilithium public keys
  • Validator selection seed generation

SHA-3 was selected over SHA-2 because its sponge construction provides a different mathematical foundation, offering defense-in-depth in the event that structural weaknesses are discovered in Merkle-Damgard-based hash functions.

2.3 Address Format

Wallet addresses are derived from Dilithium public keys using the following process:

address = "CUR" || hex(SHA3-256(dilithium_public_key)[0..20]) Total length: 3 (prefix) + 40 (hex) = 43 characters Example: CUR7f3a8b1c2d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8

The 20-byte truncation provides 160 bits of collision resistance, which is sufficient for address uniqueness while keeping addresses compact.

2.4 Wallet Encryption

Private keys are stored on disk encrypted with AES-256-GCM. The encryption key is derived from a user-supplied password using Argon2, a memory-hard key derivation function that resists GPU and ASIC-based brute-force attacks. The encrypted wallet file contains the Argon2 salt (16 bytes), AES-GCM nonce (12 bytes), and ciphertext.

3. Blockchain Architecture

3.1 Account Model

CURS3D uses an account-based state model (as opposed to UTXO). Each account is identified by its 20-byte address and stores four fields:

  • balance (u64) -- Liquid token balance in microtokens
  • nonce (u64) -- Transaction sequence number for replay protection
  • staked_balance (u64) -- Tokens locked for validator staking
  • public_key (Option) -- Dilithium public key, set on first transaction

3.2 Transactions

The protocol supports two transaction types:

  • Transfer -- Moves tokens from sender to recipient. Debits amount + fee from sender, credits amount to recipient.
  • Stake -- Moves tokens from liquid balance to staked balance. Debits amount + fee from balance, credits amount to staked_balance.

Additionally, each block contains a Coinbase transaction that credits the block reward to the block producer. Coinbase transactions have no sender and no fee.

All non-coinbase transactions must include a valid Dilithium signature over the transaction hash. The nonce must match the sender's current nonce exactly, preventing replay attacks.

3.3 Block Structure

+-------------------------------------------------------+ | Block Header | |-------------------------------------------------------| | height : u64 -- Sequential block number | | timestamp : i64 -- Unix epoch seconds | | prev_hash : [u8;32] -- SHA3 hash of parent block | | merkle_root : [u8;32] -- Merkle root of txs | | state_root : [u8;32] -- SHA3 of sorted accounts | | validator_pubkey: Vec<u8> -- Dilithium public key | +-------------------------------------------------------+ | Transactions | |-------------------------------------------------------| | [0] Coinbase { recipient, amount } | | [1] Transfer { sender, recipient, amount, fee, nonce } | | [2] Stake { sender, amount, fee, nonce } | | ... | +-------------------------------------------------------+ | Signature | |-------------------------------------------------------| | Dilithium Level 5 signature over block hash | +-------------------------------------------------------+

3.4 State Root

The state root is computed by sorting all accounts deterministically by address, serializing each account state, and computing a SHA3-256 hash over the concatenation. This enables any node to verify that it has arrived at the same global state after processing a block's transactions.

3.5 Block Validation Rules

A block is valid if and only if all of the following conditions hold:

  1. Block height is exactly one greater than the parent block height
  2. prev_hash matches the hash of the parent block
  3. Block hash matches the recomputed hash of the header
  4. Merkle root matches the Merkle tree of the block's transactions
  5. Block timestamp is between the parent timestamp and 30 seconds in the future
  6. Block signature is a valid Dilithium signature from the declared validator
  7. The validator is the selected proposer for this height (or any active validator if no selection applies)
  8. Exactly one coinbase transaction exists, with correct reward amount
  9. All non-coinbase transactions have valid signatures, correct nonces, and sufficient balances
  10. State root matches the state after applying all transactions

4. Consensus Mechanism

4.1 Proof of Stake

CURS3D uses Proof of Stake for Sybil resistance. An account qualifies as a validator by staking a minimum amount of tokens (default: 1,000 CUR = 1,000,000,000 microtokens). The validator set is determined by the current account state: any account with staked_balance >= minimum_stake and a registered public key that has not been slashed is considered active.

4.2 Validator Selection

For each block height, a single validator is selected as the block proposer using a deterministic, stake-weighted pseudo-random algorithm:

  1. Compute a seed: SHA3(block_height || prev_block_hash)
  2. Extract a 64-bit selector from the first 8 bytes of the seed
  3. Compute selector = selector % total_active_stake
  4. Iterate through validators sorted by address. For each validator, subtract its stake from the selector. The first validator where selector < stake is selected.

This construction ensures that selection probability is proportional to stake, is deterministic given the same chain state, and is unpredictable without knowledge of the next block's prev_hash.

4.3 BFT Finality

After a block is produced, validators broadcast finality votes. A FinalityVote is a Dilithium signature over block_hash || block_height. The FinalityTracker collects these votes and checks whether the cumulative staked balance of voters meets or exceeds 2/3 of the total active stake:

Finality condition: voted_stake * 3 >= total_active_stake * 2 Example with 3 validators (3000, 3000, 3000 stake each): Total active stake = 9000 Required: 6000 (2/3) Vote 1 (3000): 3000 * 3 = 9000 < 9000 * 2 = 18000 -- NOT finalized Vote 2 (3000): 6000 * 3 = 18000 >= 9000 * 2 = 18000 -- FINALIZED

Once finalized, a block and all its ancestors become permanent. The BlockTree prunes non-canonical branches at or below the finalized height, preventing long-range reorganizations.

4.4 Slashing

Equivocation -- signing two different blocks at the same height -- is a provably detectable offense. Any node that observes two valid signatures from the same validator public key on different block hashes at the same height can construct an EquivocationEvidence proof containing:

  • The block height
  • The validator's Dilithium public key
  • Two distinct block hashes with their respective valid signatures

This evidence is verified on-chain by confirming both signatures are valid Dilithium signatures and the block hashes differ. Upon successful verification, the validator's staked balance is reduced by 33%, and the validator is permanently excluded from the active set.

Equivocation evidence is broadcast to the network via the SlashingEvidence message type, allowing all nodes to apply the penalty independently.

5. Fork Choice Rule

CURS3D resolves competing chain tips using a heaviest chain by cumulative proposer-stake weight rule. The BlockTree maintains a directed acyclic graph of all known blocks. Each block entry stores its cumulative weight, defined as the sum of the proposer's stake for every block in the chain from genesis to that block.

When a new block arrives:

  1. Its parent must exist in the tree (orphan blocks are rejected)
  2. Its height must be above the current finalized height (blocks below finality are rejected)
  3. Its cumulative weight is computed as parent_weight + proposer_stake
  4. If the new block's cumulative weight exceeds the current canonical tip, the canonical tip switches (reorg)

This mechanism incentivizes validators with high stake, as their blocks carry more weight. It also provides a deterministic resolution for forks: the chain with the most cumulative validator stake always wins.

BlockTree Fork Visualization: Genesis (weight: 0) | +-- Block A1 (validator stake: 5000, weight: 5000) | | | +-- Block A2 (validator stake: 3000, weight: 8000) <-- CANONICAL TIP | +-- Block B1 (validator stake: 2000, weight: 2000) | +-- Block B2 (validator stake: 4000, weight: 6000) Chain A wins: 8000 > 6000 cumulative weight

When a block is finalized, the tree prunes all non-canonical entries at or below the finalized height. This bounds the memory and storage overhead of fork tracking.

6. Network Protocol

6.1 Transport Layer

CURS3D builds on libp2p with TCP transport, Noise protocol for authenticated encryption, and Yamux for stream multiplexing. Each node is identified by a unique PeerId derived from its libp2p identity keypair (separate from the Dilithium wallet keypair).

6.2 Message Dissemination

All protocol messages are disseminated via Gossipsub on a single topic (curs3d-mainnet). Gossipsub is configured with strict message validation, a 10-second heartbeat interval, and a 10 MB maximum message size to accommodate Dilithium signatures (which are larger than classical signatures).

6.3 Peer Discovery

Peers are discovered through mDNS for local network environments and explicit bootnode multiaddresses for wide-area connectivity. When a new peer is discovered, it is added to the Gossipsub mesh.

6.4 Sync Protocol

Chain synchronization follows this process:

  1. Validators periodically broadcast signed HeightAnnounce messages containing their current height, latest hash, and genesis hash
  2. A node receiving an announce with a greater height initiates sync by broadcasting a RequestBlocks message
  3. Peers respond with BlockResponse messages containing up to 50 serialized blocks per batch
  4. The requesting node validates and applies blocks sequentially
  5. Sync requests include genesis hash verification to prevent cross-chain contamination
  6. Sync has a 15-second timeout with up to 3 retries before resetting

Signed height announcements (containing a Dilithium signature) are treated as verified, providing stronger assurance that the announcing peer actually holds the claimed chain state. Unsigned announces still trigger sync but are logged as unverified.

6.5 Equivocation Detection

When a node receives a block at a height it already has (via the normal NewBlock message), and the block is from the same validator but with a different hash, the node automatically constructs EquivocationEvidence from the two conflicting block signatures and broadcasts it to the network.

7. Security Model

7.1 Quantum Resistance

All signature operations in CURS3D use CRYSTALS-Dilithium Level 5. The security of Dilithium relies on the hardness of the Module Learning With Errors problem, which is not known to be efficiently solvable by quantum computers. NIST selected Dilithium as the primary post-quantum signature standard (FIPS 204) after a multi-year evaluation process.

SHA-3 provides 128-bit quantum security (256/2 via Grover's algorithm), which is considered sufficient for hash function security.

7.2 Byzantine Fault Tolerance

The finality mechanism tolerates up to f < n/3 Byzantine validators, where n is the total number of active validators weighted by stake. As long as more than 2/3 of total active stake is controlled by honest validators, finalized blocks cannot be reverted.

7.3 Equivocation Accountability

Equivocation is cryptographically provable. Any observer who holds two valid signatures from the same validator on different blocks at the same height can construct and verify the evidence independently. The 33% slash penalty provides strong economic disincentive against equivocation.

7.4 Fork Choice Safety

The heaviest-chain fork choice rule ensures that an attacker would need to control more cumulative validator stake than the honest chain to override the canonical tip. Combined with finality, this prevents both short-range and long-range attacks once blocks are finalized.

7.5 Replay Protection

Each account maintains a nonce that must be incremented with each transaction. Transactions with incorrect nonces are rejected. This prevents replay of previously submitted transactions.

7.6 Timestamp Validation

Blocks must have a timestamp between the parent block's timestamp and 30 seconds in the future. This prevents timestamp manipulation while allowing for reasonable clock drift.

8. Tokenomics

8.1 Token Unit

The native token is CUR. The smallest divisible unit is the microtoken: 1 CUR = 1,000,000 microtokens. All internal accounting uses microtokens (u64).

8.2 Block Rewards

Each block awards its producer a coinbase reward. The default reward is 50 CUR (50,000,000 microtokens) per block. The reward amount is configurable via the genesis configuration.

8.3 Transaction Fees

Every non-coinbase transaction includes a fee denominated in microtokens (default: 1,000 microtokens). Fees are deducted from the sender's balance but are not currently redistributed -- they are burned, reducing circulating supply.

8.4 Staking Economics

Validators must stake a minimum of 1,000 CUR (configurable) to be eligible for block production. Selection probability is proportional to stake, creating a direct relationship between economic commitment and block production opportunity. The block reward provides the return on staked capital.

8.5 Slashing Economics

Equivocation carries a 33% penalty on staked balance, plus permanent exclusion from the validator set. This asymmetric risk (potential loss of 1/3 of stake for any attempt to double-sign) ensures that the expected value of equivocation is strongly negative.

  • CRYSTALS-Dilithium Level 5 wallet system with AES-256-GCM encryption
  • Heaviest-chain fork choice rule with BlockTree
  • Provable equivocation slashing
  • libp2p networking with Gossipsub and mDNS

10. Conclusion

CURS3D demonstrates that a blockchain protocol can be built entirely on post-quantum cryptographic primitives without sacrificing the properties expected of a modern Layer 1: deterministic finality, stake-weighted consensus, fork resolution, and efficient peer-to-peer networking.

By adopting CRYSTALS-Dilithium Level 5 and SHA-3 from inception, CURS3D avoids the substantial technical debt and upgrade risk associated with retrofitting quantum resistance onto an existing protocol. The BFT finality mechanism provides safety under the standard 2/3 honesty assumption, and the provable slashing system holds validators accountable through cryptographic evidence rather than social consensus.

The protocol is implemented in Rust and released under the MIT license. The complete source code is available at github.com/Pazificateur69/curs3d.
